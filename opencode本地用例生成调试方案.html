LEAF_PRO / OpenCode Local Authoring Plan

基于 OpenCode 的本地用例生成与调试方案

目标不是先做大型平台，而是让用户在本地打开工程后，只调用一次 /leaf-new-case；该 Command 内部启动 leaf-test-author 主流程 Skill，确认计划后自动完成编写、校验、运行、GUI 辅助调试、重跑和经验入库。

推荐主线：OpenCode Skill / Subagent 执行后端：pytest GUI 定位：调试辅助接入 经验库：内部默认入库、分层更新 日期：2026-06-23
1 主入口：leaf-test-author subagent 负责本地用户交互和生成调试闭环
4 核心分层：Command、Skill、pytest、AW/fixture
1 默认用户确认点：只确认计划；高风险和核心 AW 修改才例外确认
6 阶段路线：基础接入、工具层、调试、GUI、经验、团队化

一、方案判断

当前目标是本地生成和调试 pytest 用例，因此 OpenCode Skill / subagent 比先做完整自建 Agent 平台更直接。LangGraph 和 Orchestrator 暂不进入这条本地用例编写主链路。

推荐：OpenCode Skill / Subagent 主线

让用户直接在本地工程里输入需求，leaf-test-author subagent 读取领域插件、工具 schema 和已有测试，生成用例、运行校验、根据失败修正。它优先解决“用户如何真正用起来”。

本地优先 生成质量迭代快 用户交互自然

后置：平台化执行能力

用于后续平台化真实设备长任务、审批中断和 checkpoint/resume。当前本地用例编写阶段不需要复杂执行架构。

适合执行编排 适合 checkpoint 实现成本更高

二、用户实际使用流

用户不需要反复提醒继续流程。一次调用 /leaf-new-case 后，Command 内部交给 leaf-test-author subagent 生成 plan；用户确认后，后续编写、调试、GUI 调整、重跑和经验更新自动执行到底。确定性 Python 工具只作为 subagent 可调用的底层执行器，不作为用户主入口。

三、工程分层

OpenCode 负责本地协作体验，leaf-test-author subagent 负责推理和流程推进，LEAF Python 工具层负责确定性执行。这样可以避免把状态文件、设备探测、安全规则和报告校验写成纯 prompt。

四、用户 In Loop 设计

默认只有一个确认点：确认 plan。确认后 leaf-test-author 自动继续，不再要求用户提醒“继续编写/继续调试/保存经验”。高风险动作和核心 AW 语义修改是例外。

一次调用

用户只调用 /leaf-new-case <domain> "<teststep>"。该 Command 启动 leaf-test-author subagent；subagent 调用确定性工具创建 .leaf/runs/<run_id>/workflow.json 和 plan.json，记录当前阶段、领域、平台、确认状态和调试次数。未确认 plan 前不生成 pytest 草稿。

计划确认

Skill 展示识别出的领域、平台、目标、工具、计划步骤和将写入的文件。用户确认后调用确定性工具写入 confirmed_plan=true，进入 pytest_draft 阶段并生成 pytest 草稿，后续流程自动执行。

自动编写与校验

leaf-test-author 自动生成 pytest 文件，优先复用领域 AW/fixture，并调用轻量校验检查导入、fixture、AW 方法和平台经验引用；不通过则自动修正。

自动调试循环

pytest 失败后自动读取 traceback、pytest log、截图、控件树和 hilog，加载领域 Skill 与 GUI context，修正 pytest、fallback locator 或等待条件，并重跑直到通过或达到最大次数。

自动经验入库

内部用户默认入库。调试成功或失败都会更新本地/团队经验记录；普通 locator fallback 和等待条件可自动更新，核心 AW 语义修改只生成 review proposal。

五、GUI 介入方式

GUI 的意义是让系统根据真实页面状态实时调整用例，并把稳定修复沉淀回领域经验和 AW 映射。它不替代主流程，而是leaf-test-author 调试循环里的观察和修正能力。

页面状态识别

读取截图、控件树和 hilog，判断当前页面是目标页、权限弹窗、加载中、异常页还是控件定位失败。

自动用例调整

对低风险修复自动应用，例如插入权限处理、增加等待条件、加入 fallback locator，然后立即重跑验证。

AW / 经验更新

将成功修复写入平台化经验库；新增 fallback locator 可自动落库，修改核心 AW 方法或设备命令只生成 proposal。

六、建议命令与技能

用户只需要记住一个 slash command。复杂行为藏在 leaf-test-author subagent 里，实际执行由 pytest 完成；确定性工具层只做状态文件、用例草稿、设备探测、GUI 上下文和经验更新。

OpenCode Commands

/leaf-new-case camera "打开相机；切拍照模式；点击拍照；检查相册出现新照片" # 正常用户只需要这个入口： # 生成 workflow/plan -> 用户确认 -> 写 pytest -> pytest 执行 -> 自动调试 -> 自动入库 /leaf-resume # 读取 workflow.json 并返回下一步动作，例如 present_plan_for_confirmation 或 validate_pytest_draft

Skills

  • leaf-test-author：主流程 Skill，由 /leaf-new-case 调用，控制完整 workflow。
  • leaf-camera / leaf-audio / leaf-display / leaf-video：按领域和平台自动加载。
  • leaf-gui-context：页面状态识别、locator fallback、等待条件修正。
  • leaf-experience：自动写入经验、更新统计、生成 AW proposal。
  • tools/leaf_author：确定性工具层，被 subagent 调用，负责 workflow、plan、pytest 草稿、校验、草稿运行、只读 GUI context、经验记录、团队导出和 HDC 设备探测。

七、稳定执行机制

流程稳定性不能依赖 prompt 约束。leaf-test-author 必须把每一步写入本地状态，并通过 pytest 结果和轻量校验 gate 判断能否进入下一阶段。

workflow.json

每次任务创建 .leaf/runs/<run_id>/workflow.json，记录 current_phase、confirmed_plan、调试次数、领域、平台和产物路径。

pytest Gate

每个阶段通过确定性结果推进：domain inspect、pytest 静态检查、pytest 执行、pytest 结果摘要、GUI context 采集。失败则自动修复或停在明确阻塞点。

调试上限

自动调试循环设置最大次数。低风险修复可自动应用；高风险动作、核心 AW 语义修改和真实设备命令修改只生成 proposal。

八、经验收集

用户都是内部用户，经验默认入库或更新领域经验库。仍需做平台隔离、脱敏和质量分层，避免 OpenHarmony、Android、iOS 或不同 AW 版本的经验互相污染。

平台化目录

.leaf/knowledge/ camera/ openharmony/ aw/ experience/ templates/ android/ aw/ experience/ templates/ { "domain": "camera", "platform": "openharmony", "target_feature": "camera.capture", "failure": {"step": "camera.take_photo"}, "fix": {"type": "insert_step"}, "usage": {"success_count": 7}, "confidence": 0.87 }

自动更新策略

  • 普通失败修复、等待条件、fallback locator 默认自动入库。
  • 按 domain/platform/target_feature/AW version 区分经验。
  • confidence >= 0.85 的经验可自动用于生成用例。
  • 核心 AW 方法、真实设备命令和高风险步骤只生成 review proposal。

九、落地路线

先让用户本地可用：输入 domain + teststep，生成 pytest，跑 pytest，失败后 GUI 辅助修正，经验自动入库。复杂执行编排不进入当前阶段。

Stage A：基础接入

新增 AGENTS.md、leaf-test-author、/leaf-new-case、/leaf-resume 和 tools/leaf_author，并定义 workflow.json 状态机。Stage A 明确 plan 未确认前只生成 workflow、plan 和只读设备探测产物。

Stage B：pytest 生成

确认 plan 后生成 tests/generated/test_*.py 草稿和 Hypium 草稿，并用 validation.json 校验 run id、domain、target feature、步骤注释和元数据断言。

Stage C：调试闭环

当前本地闭环先执行草稿质量门禁，写入 pytest_result.json 并明确 DRAFT_STATIC_PASS；真实设备结论只由 Hypium 执行结果写入。

Stage D：GUI 上下文

接入 HDC 只读 uitest dumpLayout 和 hilog 采集，写入 gui_context.json；当前阶段不自动点击、不安装、不修改设备状态。

Stage E：经验沉淀

按 domain/platform 写入 .leaf/knowledge 下的 reviewable experience，标记草稿运行状态、质量门禁和不可自动应用。

Stage F：团队化

生成 team_export_manifest.json，汇总 workflow、plan、pytest、validation、pytest result 和 experience，供团队 review 后再同步。

最终结论： 当前目标下，用户只调用 /leaf-new-case <domain> "<teststep>" 一次；确认 plan 后，leaf-test-author subagent 通过 workflow.json 自动推进 pytest 草稿生成、草稿校验、草稿运行、只读 GUI context、reviewable 经验记录和团队导出。确定性 Python 工具层只负责可测试的文件读写、pytest 草稿生成、HDC 设备探测、结果摘要和本地产物，不作为用户主入口。领域 Skill 按 domain/platform 自动加载，GUI 负责根据真实页面状态提供修复上下文，经验按平台分层并先进入 reviewable 状态。自建 Agent / LangGraph 后置，不参与当前本地 pytest 用例生成主链路。
