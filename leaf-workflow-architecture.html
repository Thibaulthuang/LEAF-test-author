OpenCode 本地用例生成工作流说明

本文描述当前仓库的真实 workflow：用户从 /leaf-new-case 进入，OpenCode 技能负责语义规划，Python 工具层负责稳定产物、验证、设备探测与真实执行。

默认平台 OpenHarmony 两阶段确认 Camera 域已接入 GUI Agent 只读接入

当前 Workflow

每个请求创建一个 run 目录，所有状态和证据都落盘，断点恢复依赖 workflow.json，而不是依赖对话上下文。

1. 用户入口/leaf-new-case camera "打开相机拍照"
2. 语义计划加载 leaf-camera，生成 plan_input.json。
3. 工具落盘创建 workflow.json 和 plan.json。
4. 首次确认用户确认计划后才生成 pytest 与 Hypium draft。
5. 安全本地阶段验证 draft、跑静态/pytest gate、收集只读 GUI 上下文。
6. 真实执行第二次确认后执行 direct smoke、capture E2E 或 Hypium E2E。

用户在循环中的位置

用户不是每一步都参与。当前设计只有两个必须停下来的 checkpoint：计划确认和真实设备变更确认。

OpenCode 如何稳定触发阶段

关键不是让模型记住流程，而是让命令、技能、状态机和质量门共同限制它只能按可恢复的路径前进。

触发稳定性

  • .opencode/commands/leaf-new-case.md 固定入口，要求先调用 leaf-test-author。
  • leaf-test-author 只做 workflow 决策，不直接拼复杂文件格式。
  • 领域理解必须加载 leaf-<domain>，Camera 场景由 leaf-camera 负责语义展开。
  • 每个阶段通过 Python CLI 子命令触发：new-case、confirm-plan、advance、resume、run-e2e。
  • 工具返回 plan_summary、resume_summary、quality_gate 和 next_action，OpenCode 不需要猜下一步。

上下文稳定性

  • 对话上下文可以丢失，但 .leaf/runs/<run_id>/workflow.json 保存当前 phase。
  • 所有关键中间产物都有路径记录在 artifacts 中。
  • /leaf-resume <run_id> 读取 workflow，再调用 resume 决定安全下一步。
  • LLM 生成的语义输入落盘到 plan_input.json，后续 plan/case/draft 都能追溯。
  • 真实设备证据单独写 JSON，避免只靠终端输出判断是否通过。

任务稳定性

  • 首次确认前只允许 workflow、plan、只读 probe 产物。
  • 确认后 advance 自动跑安全本地阶段，不停在半成品。
  • 真实设备变更必须显式传入 --run-real，Camera capture 还要 --camera-capture。
  • 质量门区分草稿、框架 smoke、业务 capture、Hypium 真机通过，避免误报。
  • 失败阶段写入对应 JSON，下一次 resume 可以从失败点分析，而不是重跑高风险动作。

是否需要 Subagents

需要，但要克制。当前 leaf-test-author 可以视为主 subagent：它拥有 workflow 决策权。领域技能承担领域 subagent 的角色，但不直接改工具层。

  • 建议保留一个主编排 subagent：leaf-test-author。
  • 每个领域一个领域 subagent/skill：leaf-camera、未来 leaf-display。
  • GUI Agent 应作为证据采集/诊断 subagent 接入，默认只读。
  • 不要让多个 subagent 同时写同一个 run 状态；状态迁移仍由 Python 工具层集中执行。
Subagents 的价值是隔离职责和压缩上下文，不是增加自由度。真正保证稳定的是落盘状态、确定性工具、显式质量门和用户 checkpoint。

代码级触发链路

下面把“稳定触发”展开到实际代码。核心思想是：OpenCode 只负责按协议调用，真正的阶段边界由 Python 工具层和落盘状态控制。

1. 命令协议先限制 OpenCode

.opencode/commands/leaf-new-case.md 规定了固定顺序：加载主 skill、加载领域 skill、写 plan_input.json、调用 Python、等待用户确认。

1. Invoke the `leaf-test-author` skill/subagent.
3. Load the domain skill, such as `leaf-camera`.
4. Call deterministic tools under `tools/leaf_author/`.
5. Present the generated plan for confirmation.
6. If the user replies `yes`, call `confirm-plan`.
7. Immediately call `advance <run_id>`.
8. Stop before real-device execution.

这里不是建议，而是 OpenCode 命令的执行合同。它让模型从“自由发挥”变成“按阶段调用”。

2. CLI 只暴露固定阶段入口

tools/leaf_author/__main__.py 把动作收敛成有限子命令，OpenCode 不能传一段自然语言让工具自由解释。

new_case = subparsers.add_parser("new-case")
confirm = subparsers.add_parser("confirm-plan")
resume = subparsers.add_parser("resume")
advance = subparsers.add_parser("advance")

advance.add_argument("--run-real", action="store_true")
advance.add_argument("--camera-direct", action="store_true")
advance.add_argument("--camera-capture", action="store_true")

没有 --run-real，advance 只能跑安全本地阶段；没有 --camera-capture，不会触发真实拍照。

3. 每个 run 落盘成状态机

workflow.py 创建可恢复状态。对话上下文丢失时，/leaf-resume 读取这里，而不是让模型回忆。

workflow = {
  "owner": "leaf-test-author",
  "domain": domain,
  "platform": "openharmony",
  "current_phase": "plan",
  "confirmed_plan": False,
  "artifacts": {
    "workflow": ".leaf/runs/<run_id>/workflow.json",
    "plan": ".leaf/runs/<run_id>/plan.json"
  }
}

稳定性来自 current_phase 和 confirmed_plan。OpenCode 只需要读状态并按返回的 next_action 继续。

4. 语义计划必须通过确定性校验

planner.py 校验 OpenCode 写入的 plan_input.json。Camera capture 必须包含打开、拍照动作、新照片验证。

steps = _validated_steps(plan_input)
target_feature = str(plan_input.get("target_feature"))
_validate_domain_plan(domain, target_feature, steps)

required_groups = [
  ("打开", "相机"),
  ("拍照模式", "快门"),
  ("新照片", "照片")
]

所以即使用户只说“打开相机拍照”，OpenCode 也必须扩成完整语义步骤；扩错会被 Python 拒绝。

5. resume/advance 串起阶段

resume_run 根据 phase 返回下一步，advance_run 自动跑安全本地阶段，减少模型逐步判断的机会。

if current_phase == "plan" and not confirmed:
    next_action = "present_plan_for_confirmation"
elif current_phase in {"pytest_draft", "hypium_draft"}:
    next_action = "validate_pytest_draft"
elif current_phase == "validated":
    next_action = "run_pytest_draft"

if current_phase in {"pytest_draft", "hypium_draft"}:
    validate_pytest_draft(root, run_id)
if current_phase == "validated":
    run_pytest_draft(root, run_id)

这就是为什么确认后要求立刻 advance：安全阶段由状态机推进，不靠 OpenCode 一步步猜。

6. 质量门把结果类型固定住

camera_smoke.py 把不同失败归一化成稳定质量门，并返回说明和建议。

if "TARGET_CANDIDATES_EMPTY" in missing:
    return "CAMERA_SMOKE_TARGET_MISSING"
if "TEST_RUNNER_HAP_MISSING" in missing:
    return "CAMERA_SMOKE_TEST_RUNNER_MISSING"

return {
  "quality_gate": quality_gate,
  "quality_gate_description": ...,
  "recommended_actions": ...
}

OpenCode 不需要从 hdc 输出里猜原因。它只要展示 quality_gate、说明和下一步建议。

最终链路是：固定命令 -> skill 协议 -> plan_input.json -> Python 校验 -> workflow.json -> 用户确认 -> confirm-plan -> advance -> 显式 --run-real -> 质量门 -> resume 可恢复。

阶段稳定性保障

稳定性的核心是“LLM 只做决策和解释，确定性工具负责文件格式、状态迁移、质量门和设备命令”。

Plan
产物：.leaf/runs/<run_id>/workflow.json、plan.json。
稳定点计划由 Python 校验；未确认前不生成 draft，不执行高风险设备动作。
Confirm
产物：case.json、tests/generated/*.py、hypium/*.test.ets、OpenHarmony export。
稳定点pytest 与 Hypium 都从 case.json 派生，避免两套生成逻辑漂移。
Validate
产物：validation.json、pytest_result.json。
质量门DRAFT_VALIDATED 和 DRAFT_STATIC_PASS 只代表 draft 可执行，不代表真实设备通过。
GUI Context
产物：gui_context.json，包含 layout 和 hilog 摘要。
稳定点当前 GUI Agent 只读，不点击、不修复、不改 AW 代码。
Real Device
产物：camera_direct_smoke.json、camera_capture_e2e.json、hypium_result.json。
安全边界拍照会创建媒体文件，必须在计划确认后再二次确认。
Experience
产物：.leaf/knowledge/<domain>/<platform>/experience/*.json、team_export_manifest.json。
稳定点经验记录是 reviewable draft knowledge，不自动修改 AW 或业务代码。

领域插件如何插入

领域插件不是 Python 包，而是 OpenCode skill + Python deterministic mapping 的组合。当前 Camera 是参考实现。

1. OpenCode Skill

新增 .opencode/skills/leaf-<domain>/SKILL.md，定义自然语言如何展开成领域动作、默认平台、风险和真实设备边界。

domain = "camera"
target_feature = "camera.capture"
steps = [
  "打开系统相机",
  "确认处于拍照模式",
  "点击快门拍照",
  "检查产生新照片"
]

2. Plan 与 Case Spec

在 planner.py 和 case_spec.py 中加入领域验证和动作映射。这里的规则必须是确定性的，可测试的。

CameraAW.launch
CameraAW.switchToPhotoMode
CameraAW.capture
GalleryAW.assertLatestPhotoCreatedAfter

3. 生成与执行器

在 hypium.py、领域 smoke/e2e 模块中加入 draft 渲染、AW stub、preflight 和质量门。新增领域必须配套测试。

quality_gate:
  CAMERA_DIRECT_SMOKE_PASS
  CAMERA_CAPTURE_E2E_PASS
  HYPIUM_REAL_PASS
插件接入原则：OpenCode 负责语义理解，Python 工具层负责格式、状态、设备命令和可重复测试。不要把确定性 JSON 格式只写在 prompt 里。

GUI Agent 怎么接入

当前 leaf-gui-agent 是预留的只读调试代理，已通过 collect_gui_context 接入 workflow。

当前允许

  • 读取 HDC target metadata。
  • 执行 uitest dumpLayout 获取布局。
  • 执行 hilog -x 获取日志。
  • 写入 gui_context.json，供后续人工或 agent 分析。

下一步扩展

  • 增加 screenshot pull 和布局节点索引。
  • 生成 locator fallback proposal，但先只写 reviewable 建议。
  • 处理权限弹窗前必须有 workflow 状态和用户确认。
  • 自动点击、等待、修复 AW 都必须单独引入质量门。
